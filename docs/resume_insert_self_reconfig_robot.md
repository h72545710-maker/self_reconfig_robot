## 面试时一句话介绍

我做的是一个面向自重构机器人的多模块协同控制原型，不是单车路径规划，而是把多个机器人模块组织成可合体、可分体的系统；上位机负责任务调度、leader 选举、路径规划、故障恢复和 ROS2 状态发布，下位机接口预留给 STM32 执行电机控制和传感器采集。

## 面试时 1 分钟讲法

这个项目主要解决多模块机器人在任务执行中的协同控制问题。我把每个机器人单体抽象成一个模块，系统支持 `coupled` 合体编队和 `split` 分体并行两种模式：合体时由 leader 带队，follower 根据队形目标独立规划；分体时把探索任务拆成多个子任务并行执行。控制过程中实现了 leader 动态选举、故障恢复、低电量让渡、动态障碍重规划和协同感知覆盖。

算法上，我用 A* 做栅格路径规划，再用 line-of-sight 做航点平滑，减少下发给执行层的控制点；状态估计上模拟融合里程计、视觉定位、IMU 航向和前向距离，输出置信度和误差。工程上，我实现了 UDP 心跳、ACK、任务下发和状态回传，并在树莓派上通过 Docker 跑 ROS2 Jazzy，把控制状态拆成多个 ROS2 topic 发布，形成从 C++ 控制仿真到 ROS2 订阅验证的闭环。

## 面试官可能追问

### 1. 这个项目和普通路径规划有什么区别？

普通路径规划主要关注单个机器人从起点到终点怎么走；这个项目重点是多个模块如何组织起来完成任务，包括合体/分体模式切换、leader 选举、任务拆分、故障恢复、低电量让渡和协同感知覆盖。

### 2. 哪里体现自重构？

体现在工作模式和组织形态上。系统不是固定单车，而是支持多个模块合体形成整体队形，也支持拆分成多个单体并行执行子任务；子任务完成后再重新回到合体模式。

### 3. ROS2 做到了什么程度？

当前没有完全迁移到 ROS2 Navigation2，而是实现了 ROS2 bridge：C++ 控制核心仍独立运行，状态通过 JSON 文件桥接到 ROS2，并拆分发布为 mission、modules、path、metrics、events 等 topic。这样既保留底层控制实现，也能接入 ROS2 的 topic、launch、rosbag、RViz 等生态。

### 4. 真实硬件怎么接？

树莓派运行上位机控制层和 ROS2 bridge，STM32 负责电机闭环和传感器采集。当前项目预留了 `stm32_bridge`，可以通过 UART 接收 `ODOM/BAT/FAULT`，并把故障、电量和里程计反馈给上位机控制层。

### 5. 当前不足是什么？

目前主要完成的是控制层原型和硬件可接入接口，真实机械自重构结构、复杂运动学模型和实体多模块联调还没有完成。后续可以把 String topic 升级为 ROS2 typed message，并接入真实编码器、IMU、超声波或 ToF 传感器。
