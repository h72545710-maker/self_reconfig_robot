# 简历项目版本：自重构机器人多模块协同控制系统

## 项目名称

自重构机器人多模块协同控制与硬件可接入原型系统

## 简历描述

基于 C++17 实现面向自重构机器人的多模块协同控制原型，设计合体/分体两种工作模式，支持多阶段任务调度、leader 动态选举、故障恢复、低电量让渡、动态障碍重规划和多模块协同感知覆盖；构建 Web 可视化界面展示模块状态、路径规划、任务时间轴和控制指标，并预留 ROS2 与 STM32 下位机接入能力。

## 简历 Bullet 版本

- 使用 C++17 设计多模块机器人控制框架，将机器人单体抽象为独立模块，支持 `coupled` 合体编队和 `split` 分体并行任务两种模式，实现“集合 -> 探索 -> 返航”的多阶段任务流程。
- 实现 leader 动态选举与健康管理机制，综合模块电量、目标距离和在线状态进行 leader 选择，在模块故障或低电量场景下完成控制权切换、任务重分配和故障模块绕行。
- 实现 A* 路径规划与 line-of-sight 航点平滑优化，支持动态障碍注入后的在线重规划，并将其他模块和故障模块纳入动态避障约束。
- 设计多模块协同感知与状态估计模块，模拟融合里程计、视觉定位、IMU 航向和前向距离信息，输出定位置信度、估计误差和感知覆盖率等指标。
- 搭建 UDP 通信与 ACK 可靠控制链路，支持模块注册、心跳、任务下发、状态回传和故障上报；增强 `stm32_bridge`，可通过 UART 接收 STM32 回传的 `ODOM/BAT/FAULT` 数据并联动 master 节点。
- 新增 ROS2 适配包，将控制状态桥接到 `/self_reconfig/control_state`，并拆分发布 `/self_reconfig/mission`、`/self_reconfig/modules`、`/self_reconfig/path`、`/self_reconfig/metrics`、`/self_reconfig/events` 等语义 topic，为后续接入 RViz、rosbag、导航栈或真实机器人平台提供迁移接口。
- 构建 Web 可视化界面，实时展示模块地图、leader/follower 状态、优化路径、队形目标、子任务进度、传感器覆盖、模式时间轴和关键控制指标，便于演示和复盘。

## 面试 1 分钟讲法

我这个项目做的是一个自重构机器人的上位机协同控制原型。它不是单个机器人走迷宫，而是把多个机器人模块看成可以合体也可以分体的系统：合体时由 leader 带队，follower 维持队形；分体时把探索任务拆成多个子任务并行完成。过程中我实现了 leader 动态选举、故障恢复、低电量让渡、动态障碍重规划和协同感知覆盖。

底层算法上，路径规划用 A* 保证可达，再用 line-of-sight 做航点平滑，减少下发给执行层的控制点。状态估计上，我单独抽了传感器融合层，模拟融合里程计、视觉定位、IMU 航向和前向距离，输出置信度和误差。工程上我做了 UDP 通信、ACK、心跳和任务下发，也预留了 ROS2 包和 STM32 bridge，后续可以把虚拟模块替换成真实树莓派 + STM32 下位机。

## 面试官追问时的关键词

- 自重构特点：合体/分体模式切换，而不是固定单车控制。
- 多模块协同：leader-follower 编队、分体子任务、协同覆盖。
- 鲁棒性：故障检测、leader 重选、低电量让渡、动态重规划。
- 嵌入式 Linux：C++17、UDP、串口桥接、树莓派部署、状态机。
- ROS2：当前是适配包，先发布控制状态，后续可拆 topic/action。
- 下位机：STM32 负责电机闭环和传感器采集，上位机负责任务和路径决策。

## 简历中不要过度写的点

- 不要写“已经完成真实自重构机器人整机”，因为当前主要完成的是控制原型和硬件接入接口。
- 可以写“预留/实现 STM32 bridge 接入能力”，不要写“已经完成复杂物理自重构结构”。
- ROS2 可以写“新增 ROS2 适配包”，不要写“完整迁移到 ROS2 Navigation2”。
