# Control Design Notes

本项目当前主线先聚焦上位机控制层，而不是下位机电机闭环。

## 已实现控制能力

### 1. Leader 路径规划

`robot_sim` 使用栅格地图和 A* 搜索，从当前 leader 位置规划到目标点。

规划时会把其他模块的位置作为动态占用格，避免 leader 直接穿过其他模块。

### 2. 路径航点优化

A* 原始路径通常包含每一步的格子点。控制执行时不一定需要每个点都作为航点，因此 `GridMap::optimize_path()` 会删除连续同方向的中间点，只保留转向点和终点。

运行时会输出：

```text
raw_path=19 optimized_waypoints=6 saved_waypoints=13
```

含义：

- `raw_path`：A* 原始路径长度
- `optimized_waypoints`：压缩后的控制航点数
- `saved_waypoints`：减少的中间航点数量

### 3. 多模块队形协同

leader 每次移动后，follower 会根据 leader 当前运动方向生成队形目标：

- 第一个 follower 跟在 leader 后方
- 后续 follower 分布在后方左右侧
- 如果目标格不可达或被占用，会搜索附近可用格

每个 follower 会独立规划到自己的队形目标，并把已经预定的位置加入动态避碰表。

### 4. 故障模块动态避障

故障模块不会从地图中消失，而是继续占据原位置。新 leader 被选出后，会把故障模块当作动态障碍重新规划路径。

这对应真实自重构机器人中的一个关键问题：某个模块失效后，系统不仅要重新选 leader，还要考虑失效模块对空间通道的阻塞影响。

### 5. 协同感知覆盖

每个在线模块拥有一个简化的局部传感半径。模块在运动过程中会把周围可通行栅格标记为已感知区域，Web 面板用浅绿色展示覆盖范围。

系统会统计：

- 已覆盖栅格数量
- 全局覆盖率
- 覆盖率里程碑事件

这让仿真从单纯导航扩展为多模块协同探索：leader 负责任务推进，follower 不仅保持队形，也参与环境覆盖。

### 6. 合体/分体工作模式

为了体现自重构机器人“多个单体既可以组合协同，也可以拆分执行”的特点，控制层加入了两种工作模式：

- `coupled`：合体/编队模式。多个模块围绕 leader 保持队形，作为一个整体完成集结、穿越、返航等任务。
- `split`：分体并行模式。系统把探索任务拆成多个局部子任务，模块 2、3、4 分别前往不同目标点，完成后再回到整体任务流程。

当前三阶段任务含义：

1. `RENDEZVOUS`：模块先以合体/编队模式完成集结。
2. `EXPLORE`：模块进入分体模式，并行完成多个局部探索子任务。
3. `RETURN`：子任务完成后重新回到合体/编队模式返航。

Web 面板会显示 `工作模式` 和 `子任务` 完成度，橙色框表示分体子任务目标。

同时，Web 面板提供模式时间轴，记录关键阶段：

```text
coupled 集结 -> split 并行探索 -> coupled 返航
```

时间轴还会标记模块故障、动态障碍、低电量让渡和任务完成节点，便于最终截图中直接说明系统经历过合体、分体和重新合体的过程。

## 面试表达

可以这样讲：

我把多模块系统拆成了三个控制层次：

1. 全局层：leader 基于 A* 做目标路径规划。
2. 协同层：follower 根据 leader 运动方向生成队形目标，形成 leader-follower 协同。
3. 安全层：把在线模块、故障模块都纳入动态障碍，执行逐步避碰和故障后的重规划。

这个版本还没有追求复杂物理动力学，而是先验证多模块协同控制中的路径规划、队形保持、故障恢复和动态避障逻辑。

## 可视化运行

终端控制仿真：

```powershell
.\build\Release\robot_sim.exe
```

Web 控制可视化：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\run_control_visual.ps1
```

页面地址：

```text
http://127.0.0.1:8081/web/index.html?source=control_state.json
```

可视化面板会展示模块位置、故障模块、障碍物、优化航点、队形目标、路径优化指标和事件日志。

## 复盘指标

Web 面板右侧会输出控制复盘指标：

- `重规划`：控制循环中成功生成路径的次数。
- `Leader切换`：故障恢复过程中 leader 重新选举次数。
- `队形切换`：`wedge`、`column`、`line` 队形模式切换次数。
- `轨迹长度`：leader 实际移动的栅格步数。
- `恢复Tick`：故障注入到新 leader 接管之间消耗的仿真 tick。
- `覆盖率`：多模块传感覆盖的可通行区域比例。
- `覆盖格`：已经被在线模块感知过的栅格数量。
- `工作模式`：当前处于合体协同还是分体并行执行。
- `子任务`：分体探索阶段的局部任务完成数量。

这些指标用于面试展示时说明系统不是只完成了动画效果，而是能量化多模块协同控制、故障恢复和动态重规划过程。
