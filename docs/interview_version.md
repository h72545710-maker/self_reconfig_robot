# 面试版项目讲解：自重构机器人多模块协同控制系统

## 1. 一句话介绍

我做的是一个面向自重构机器人的多模块协同控制原型。系统把多个机器人单体抽象成独立模块，支持合体编队、分体并行探索、leader 重选、故障恢复、低电量让渡、动态障碍重规划和协同感知覆盖，并通过 Web 页面可视化整个控制过程。

## 2. 项目背景

自重构机器人不是一个固定形态的机器人，而是由多个可独立工作的模块组成。它的关键问题不是单个电机怎么转，而是多个模块如何组织起来完成任务：

- 什么时候多个模块应该合体协同？
- 什么时候可以拆开并行执行？
- leader 故障后谁接管？
- 低电量模块是否应该让渡控制权？
- 动态障碍出现后如何重规划？
- 多模块探索时如何提升环境覆盖率？

所以这个项目重点实现的是上位机控制层，而不是机械结构或下位机电机闭环。

## 3. 系统架构

项目主要分为三层：

```text
Web 可视化层
  显示地图、模块状态、任务阶段、工作模式、时间轴和复盘指标

控制决策层 C++17
  路径规划、队形控制、任务调度、leader 选举、故障恢复、分体任务

通信/执行抽象层
  UDP module_node 仿真，后续可替换为树莓派 + STM32 下位机
```

当前主要程序：

- `robot_sim`：控制算法和任务仿真。
- `master_node`：UDP 多节点通信、注册、心跳、ACK、任务分配。
- `module_node`：模块节点仿真。
- `web/index.html`：控制过程可视化。

## 4. 核心控制逻辑

### 4.1 合体/分体工作模式

系统支持两种工作模式：

- `coupled`：合体/编队模式，多个模块围绕 leader 形成整体队形，适合集结、通过障碍区、返航。
- `split`：分体并行模式，把探索任务拆成多个局部子任务，不同模块分别去完成。

当前演示任务是三阶段：

```text
RENDEZVOUS 集结 -> EXPLORE 分体探索 -> RETURN 重新合体返航
```

这体现了自重构机器人的特点：多个单体既可以组合成整体工作，也可以拆开分别执行任务。

### 4.2 Leader 选举与让渡

leader 不是固定的，而是根据模块状态动态选出来。当前评分考虑两个因素：

- 电量越高越适合当 leader。
- 距离当前目标越近越适合接管。

当 leader 故障时，系统会重新选举；当 leader 电量过低时，会触发健康让渡，让其他模块接管。

### 4.3 路径规划与重规划

leader 使用 A* 在栅格地图上规划路径。规划时会把以下内容当作动态障碍：

- 静态障碍物。
- 故障模块。
- 其他模块当前占用的位置。
- 运行过程中新增的动态障碍。

同时对 A* 原始路径做航点压缩，只保留转向点和终点，减少控制执行点。

### 4.4 队形控制

follower 根据 leader 的运动方向生成队形目标，并独立规划到对应位置。

队形模式会根据场景切换：

- `wedge`：开阔区域的楔形队形。
- `column`：狭窄或障碍较多区域的纵队。
- `line`：接近目标或返航时的横向队形。

### 4.5 协同感知覆盖

每个在线模块有一个简化传感半径。模块移动时会覆盖周围可通行栅格，系统统计覆盖率和覆盖格数量。

这让分体探索不只是“模块散开”，而是能量化地提升环境感知覆盖。

## 5. 可视化演示怎么讲

演示时可以按这条线讲：

1. 初始状态下，模块处于 `coupled` 模式，先完成集结。
2. 1 号模块故障，系统检测到后重新选 leader。
3. 进入 `split` 模式，模块 2/3/4 分别执行局部探索子任务。
4. 中途新增动态障碍，触发路径重规划。
5. leader 电量过低，触发健康让渡。
6. 子任务完成后，系统重新进入 `coupled` 模式返航。
7. 最终任务完成，右侧指标显示整个过程的复盘结果。

右侧重点指标：

```text
阶段：3/3
工作模式：coupled / split
子任务：3/3
Leader切换：2
健康让渡：1
队形切换：若干次
重规划：若干次
覆盖率：约 70%+
```

模式时间轴重点讲：

```text
T0   coupled：合体集结
T14  split：分体并行探索
T31  coupled：重新合体返航
```

## 6. 项目亮点

1. 不是单机器人路径规划，而是多模块组织与任务重构。
2. 显式实现了 `coupled/split` 两种自重构工作模式。
3. 支持 leader 故障恢复和低电量健康让渡。
4. 支持动态障碍注入和在线重规划。
5. follower 不只是跟随，而是根据队形目标独立规划。
6. 多模块探索带来感知覆盖率统计。
7. Web 页面能展示过程、时间轴和复盘指标，便于面试演示。

## 7. 面试官可能追问

### Q1：这个项目和普通路径规划有什么区别？

普通路径规划通常只考虑单个机器人从起点到终点。我这个项目关注的是多个模块如何组织起来完成任务，包括合体编队、分体执行、leader 选举、故障恢复、低电量让渡和任务阶段切换。

### Q2：哪里体现了自重构？

体现在工作模式切换。系统不是固定一个机器人形态，而是支持：

- `coupled`：多个模块组合成一个整体执行任务。
- `split`：模块拆开，各自完成局部任务。
- 子任务完成后再回到 `coupled`，重新形成整体队形。

### Q3：leader 怎么选？

当前使用一个轻量评分函数，综合考虑模块电量和到目标距离。电量高、距离目标近的模块更适合接管 leader。后续可以扩展加入通信质量、负载、传感器健康状态等因素。

### Q4：故障恢复怎么做？

故障模块会被标记为 `fault`，不再参与 leader 选举，同时它的位置仍然被视为动态障碍。系统重新选 leader 后继续任务，并在路径规划时绕开故障模块。

### Q5：为什么不用 ROS2？

这个项目早期我想先把底层机制自己实现清楚，比如协议、心跳、ACK、重传、状态同步和故障恢复。后续如果迁移到 ROS2，可以把模块状态做成 topic，把任务分配做成 action，把 leader 选举和调度作为独立 node。

### Q6：如果接真实硬件怎么做？

树莓派运行上位机控制层，STM32 作为下位机控制电机和采集传感器。上位机通过 UART/CAN 下发速度、模式或任务命令，STM32 回传电机状态、电量、传感器数据。当前仿真的 `module_node` 可以替换为真实硬件桥接节点。

## 8. 可以写进简历的一段

基于 C++17 实现自重构机器人多模块协同控制原型，设计合体/分体两种工作模式，支持多阶段任务调度、A* 路径规划与航点优化、leader 选举、故障恢复、低电量健康让渡、动态障碍重规划和多模块感知覆盖统计；构建 Web 可视化界面展示模块状态、任务时间轴、子任务完成度和复盘指标，验证多模块系统在故障和环境变化下的协同控制能力。

## 9. 当前不足与后续扩展

当前项目还是控制层仿真，物理运动和传感器模型比较简化。后续可以扩展：

- 接入树莓派 + STM32 小车。
- 使用编码器/IMU/超声波做真实状态反馈。
- 将 UDP 通信替换为 UART/CAN/ROS2。
- 引入更真实的运动学模型。
- 增加任务优先级和多任务队列。
- 增加历史回放和实验数据导出。
